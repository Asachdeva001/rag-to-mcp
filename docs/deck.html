The Framework

R.I.C.E

Four elements — applied at every layer of the stack

ElementQuestion it answersCommon failure
R — RoleWho is the AI acting as?Too vague: "you are a helpful assistant"
I — IntentWhat does correct output look like — verifiably?Not verifiable: "do a good job"
C — ContextWhat information is allowed? What is excluded?No exclusions stated — AI uses anything
E — EnforcementWhat rules must never be violated?Generic: "be accurate" — not testable
"Enforcement is the most important element. It is the difference between a prompt and a system you can trust."

Slide 8 · Block 0
R.I.C.E Deep Dive

Testable vs Wishful Enforcement

A rule names a condition and a required output. A wish names a quality.

❌ Wishes — not enforceable
  • "Be accurate"
  • "Don't hallucinate"
  • "Be consistent"
  • "Stay on topic"
✅ Rules — testable
  • "Priority must be Urgent if description contains: child, school, injury, hospital"
  • "Category must be exactly one of: [fixed list]. No variations."
  • "If no chunk scores above 0.6 — return refusal template exactly"
  • "Tool description must name the three documents and refusal condition"
"If you cannot write a test that would fail when the rule is broken — it is not a rule. It is a wish."

The Scenario

The Classifier Problem

The Director dispatches two teams. There were seven Urgent complaints. The classifier missed four.

  • The system: AI classifier reads city complaint CSVs → outputs category, priority, reason, flag
  • The failure: Naive prompt returns inconsistent category names, misses severity signals, gives confident answers on ambiguous inputs
  • The consequence: Director's dashboard shows 3 Urgent. There were 7. One involved a child injury near a school.
  • Your task: Apply R.I.C.E Enforcement to make the classifier deterministic and auditable
"The output looked complete and confident. Nobody caught it because confident is not the same as correct."
Slide 14 · Block 1 · UC-0A
What Breaks

Five Failure Modes

1
Taxonomy drift
Same complaint type gets different category names across rows — "Road Infrastructure", "Pothole Issue", "Road Damage" for the same problem
2
Severity blindness
"Child fell near school" → Priority: Standard. No keyword rule = no Urgent signal
3
Missing justification
No reason field in output — Director cannot audit the decision
4
Hallucinated sub-categories
AI outputs "Pedestrian Safety Incident" — not in the allowed schema
5
False confidence on ambiguity
Ambiguous complaint classified confidently with no NEEDS_REVIEW flag

Slide 18 · Block 2 · RAG Primer
The Problem

Why LLMs Fail on Your Data

Knowledge cutoff

LLMs are trained on data up to a fixed date. Your HR policy updated last month is invisible to them.

Private data blindness

Your CMC HR, IT, and Finance policies were never public. The LLM has never seen them.

Context window limits

You cannot paste all documents into every prompt. Irrelevant text in context degrades answer quality.

"RAG is not about making the LLM smarter. It is about giving it the right context — and nothing else."

Slide 21 · Block 2 · RAG Primer
Failure Mode 1

Why Chunking Strategy Matters

Clause 5.2: "LWP requires approval from the Department Head AND the HR Director"

❌ Fixed-size (400 chars)
Chunk 11: "...LWP requires approval
from the Department Head and the"

Chunk 12: "HR Director. Manager alone
is not sufficient..."

→ Retrieval returns chunk 11
→ Answer: "Department Head"
→ HR Director dropped silently
✅ Sentence-aware
Chunk 11: "LWP requires approval from
the Department Head and the HR
Director. Manager approval
alone is not sufficient."

→ Both conditions in same chunk
→ Answer: "Dept Head AND HR Director"
→ Complete obligation preserved
"Your Enforcement rule: chunk size must not exceed 400 tokens. Never split mid-sentence."

Slide 23 · Block 2 · RAG Primer
Failure Mode 3

Answer Outside Retrieved Context

The LLM still has its training knowledge — Enforcement keeps it out

❌ No grounding enforcement

"Yes, you can use your personal phone for approved work tools and email. As is standard practice in government organisations, employees are expected to maintain appropriate security on personal devices."

↑ "standard practice" — not in any document

✅ Grounding enforcement applied

"Personal devices may be used to access CMC email and the CMC employee self-service portal only. [Source: policy_it_acceptable_use.txt, chunk 4]"

↑ Every word traceable to retrieved chunk

Slide 24 · Block 2 · RAG Primer
Summary

The Three RAG Failure Modes

All caused by missing Enforcement rules — all fixable in agents.md

1
Chunk boundary failure
Fixed-size chunking splits multi-condition clauses. Answer drops one condition silently. Fix: sentence-aware chunking + 400 token limit.
2
Wrong chunk retrieval
No threshold — wrong document's chunks retrieved. Fix: similarity threshold 0.6 + metadata filter in Enforcement.
3
Answer outside retrieved context
LLM adds training knowledge. Fix: "Answer must use only information present in retrieved chunks."
"All three failures are invisible without a systematic check. Your agents.md Enforcement rules are the check."

Slide 34 · Block 3 · MCP Primer
The Problem

Why Agents Need a Standard Tool Interface

The integration problem

N agents × M tools = custom integrations. Every agent needs to know the exact API of every tool it might call.

The discovery problem

An agent cannot ask "what tools exist?" without a standard. It has to be told explicitly — hardcoded every time.

MCP: Model Context Protocol

A standard: MCP servers expose tools. MCP clients (agents) discover and call them. Any agent can call any MCP server without custom integration.

"MCP is to AI tools what HTTP is to web APIs — a standard that makes every tool callable by every client."

Slide 36 · Block 3 · MCP Primer
The Protocol

Two JSON-RPC Methods

tools/list — discovery
Request:
{"method": "tools/list", "id": 1}

Response:
{"result": {
"tools": [{
"name": "query_policy_documents",
"description": "...", ← enforcement
"inputSchema": { ... }
}]
}}
tools/call — execution
Request:
{"method": "tools/call",
"params": {
"name": "query_policy_documents",
"arguments": {
"question": "Who approves LWP?"
}}}

Response:
{"result": {
"content": [{"type": "text", "text": "..."}],
"isError": false}}
"All responses return HTTP 200. Application errors use isError: true in the JSON body — not HTTP error codes."
