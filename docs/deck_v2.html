R.I.C.E
Role · Intent · Context · Enforcement — the framework for every prompt you write
The Framework

What is R.I.C.E?

Four elements that turn a vague instruction into a system the AI cannot misinterpret

ElementQuestion it answersWithout it
R — RoleWho is the AI acting as?AI picks a default persona — usually too generic
I — IntentWhat does correct output look like — verifiably?AI optimises for plausibility, not correctness
C — ContextWhat is the AI allowed to use? What is excluded?AI uses anything — training data, assumptions, guesses
E — EnforcementWhat specific conditions must never be violated?AI makes decisions you didn't know it was making
"Enforcement is the most critical element. It is the only one the AI will never self-correct. Where you leave a gap — it makes a decision."
R.I.C.E · Element 1 of 4
R
Role
Define who the AI is acting as — its identity, domain, and operational boundary. A specific role constrains the AI's frame of reference before it generates a single token.
Example — Complaint Classifier
❌ Weak: "You are a helpful assistant."

✅ Strong: "You are a civic complaint classification engine
           for City Municipal Corporation operations.
           You classify incoming complaints only.
           You do not answer general questions."
R.I.C.E · Element 2 of 4
I
Intent
Describe what a correct output looks like — verifiably. Not "do a good job" but a specification a human could check against the output and confirm was met.
Example — Complaint Classifier
❌ Weak: "Classify the complaint accurately."

✅ Strong: "For each complaint row, output exactly four fields:
           category, priority, reason, flag.
           Output must be deterministic —
           same input must produce same output every run."
R.I.C.E · Element 3 of 4
C
Context
Specify what information sources the AI may use — and explicitly state what it must not use. Most prompts define what is allowed. The exclusions are equally important.
Example — Complaint Classifier
❌ Weak: "Use the complaint data."

✅ Strong: "Use only the complaint description and location fields
           from the provided CSV row.
           Do not use: general knowledge, news, assumptions
           about the city, or information from other rows."
R.I.C.E · Element 4 of 4
E
Enforcement
Specific, testable rules that must never be violated. Not qualities — conditions. A rule names a trigger and a required output. If you cannot write a test that fails when the rule is broken, it is not a rule.
Example — Complaint Classifier
❌ Wish: "Be accurate and consistent."

✅ Rule: "category must be exactly one of:
            Pothole, Flooding, Streetlight, Waste, Noise,
            Road Damage, Heritage Damage, Heat Hazard,
            Drain Blockage, Other. No variations.
            priority = Urgent if description contains:
            child, school, injury, hospital, fell, collapse."

The Problem

Why LLMs Fail on Your Data

Knowledge cutoff

LLMs are trained up to a fixed date. Your HR policy updated last month, your IT security rules, your finance reimbursement limits — all invisible.

Private data blindness

Your CMC policy documents were never public. The model has never seen them. Asking it to answer from them without providing them produces confident fabrication.

Context window limits

Pasting all documents into every prompt is not a solution. A 50-page policy corpus fills the context. Irrelevant text degrades answer quality significantly.

"RAG does not make the LLM smarter. It gives the LLM exactly the right context — retrieved at query time — and enforces that it answers from that context only."

Wrong chunk retrieval
No similarity threshold — wrong document's chunks returned. "Personal phone" query retrieves HR leave policy instead of IT policy.

LLM enriches answer with training knowledge. Adds "as is standard practice" — not in any document.
Step 1 + 2 — Ingest & Chunk

Load Documents + Sentence-Aware Chunking

Never split mid-sentence. Max 400 tokens per chunk.

# chunk_documents — from stub_rag.py import os, re def chunk_documents(docs_dir: str, max_tokens: int = 400) -> list: results = [] for fname in sorted(os.listdir(docs_dir)): if not fname.endswith('.txt'): continue text = open(os.path.join(docs_dir, fname), encoding='utf-8').read() # Split on sentence boundaries — never mid-sentence sentences = re.split(r'(?<=[.!?])\s+', text.strip()) chunks, current, count = [], [], 0 for sentence in sentences: words = len(sentence.split()) if count + words > max_tokens and current: results.append({ 'doc_name': fname, 'chunk_index': len([r for r in results if r['doc_name']==fname]), 'text': ' '.join(current), }) current, count = [sentence], words else: current.append(sentence); count += words if current: results.append({'doc_name': fname, 'chunk_index': len([r for r in results if r['doc_name']==fname]), 'text': ' '.join(current)}) return results
Step 3 + 4 — Embed & Store

Vectorise with sentence-transformers + Store in ChromaDB

all-MiniLM-L6-v2 converts each chunk into 384 numbers. ChromaDB stores them on disk.

# build_index — from stub_rag.py import chromadb from sentence_transformers import SentenceTransformer def build_index(docs_dir: str, db_path: str = './chroma_db'): model = SentenceTransformer('all-MiniLM-L6-v2') # 384-dim vectors chunks = chunk_documents(docs_dir) client = chromadb.PersistentClient(path=db_path) try: client.delete_collection('policy_docs') # reset if exists except: pass collection = client.create_collection('policy_docs') ids = [c['doc_name'] + '::chunk_' + str(c['chunk_index']) for c in chunks] texts = [c['text'] for c in chunks] metadatas = [{'doc_name': c['doc_name'], 'chunk_index': c['chunk_index']} for c in chunks] embeddings = model.encode(texts, show_progress_bar=True).tolist() collection.add(ids=ids, documents=texts, metadatas=metadatas, embeddings=embeddings) print(f'Index built: {len(chunks)} chunks stored at {db_path}')
Step 5 — Retrieve

Query → Embed → Similarity Search → Threshold Filter

Chunks scoring below 0.6 are excluded — not passed to the LLM.

# retrieve — core of retrieve_and_answer in stub_rag.py def retrieve(query: str, collection, model, top_k=3, threshold=0.6): # 1. Embed the query into the same vector space as the chunks q_embedding = model.encode([query]).tolist() # 2. ChromaDB finds the top_k closest chunks by cosine similarity results = collection.query( query_embeddings=q_embedding, n_results=top_k, include=['documents', 'metadatas', 'distances'] ) docs, metas, dists = (results['documents'][0], results['metadatas'][0], results['distances'][0]) # 3. Convert L2 distance → similarity score. Filter below threshold. # distance=0 → similarity=1.0 (identical) # distance=2 → similarity=0.0 (opposite) passing = [ (doc, meta, round(1 - dist/2, 3)) for doc, meta, dist in zip(docs, metas, dists) if (1 - dist/2) >= threshold # R.I.C.E Enforcement rule ] return passing # list of (text, metadata, score)
Step 6 — Generate

Grounded Prompt → LLM → Cited Answer

If no chunk passes threshold — return the refusal template. Never guess.

# retrieve_and_answer — full function from stub_rag.py def retrieve_and_answer(query: str, collection, model, llm_call, threshold=0.6) -> dict: passing = retrieve(query, collection, model, threshold=threshold) # Enforcement: if nothing passes — refuse. Never hallucinate. if not passing: sources = ', '.join(f"{m['doc_name']}::chunk_{m['chunk_index']}" for _, m, _ in zip(docs, metas, dists)) return { 'answer': f'This question is not covered in the retrieved policy documents. ' f'Retrieved chunks: {sources}. Please contact the relevant department.', 'refused': True, 'cited_chunks': [] } # Build context — retrieved chunks ONLY. No outside knowledge. context = '\n\n'.join( f"[{m['doc_name']}, chunk {m['chunk_index']}, score={score}]:\n{doc}" for doc, m, score in passing ) prompt = (f'Answer using ONLY the context below. ' f'Cite source document and chunk index for every claim.\n\n' f'Context:\n{context}\n\nQuestion: {query}\n\nAnswer:') return {'answer': llm_call(prompt), 'refused': False, 'cited_chunks': [{'doc': m['doc_name'], 'chunk': m['chunk_index'], 'score': s} for _, m, s in passing]}
Connecting to the LLM

llm_adapter.py — Gemini + Grok

One swappable function. Change the provider without touching RAG logic.

Gemini free tier (default)

Grok free tier (alternative)
# pip3 install openai # export XAI_API_KEY="your-key" # Get free key: console.x.ai import os from openai import OpenAI def call_llm(prompt: str) -> str: client = OpenAI( api_key=os.environ['XAI_API_KEY'], base_url='https://api.x.ai/v1') r = client.chat.completions.create( model='grok-3-mini', messages=[{'role':'user', 'content':prompt}]) return r.choices[0].message.content
End-to-End

Putting It All Together

One query in. One cited answer out. Or a clean refusal.

# Full RAG flow — run from CLI from sentence_transformers import SentenceTransformer import chromadb from llm_adapter import call_llm model = SentenceTransformer('all-MiniLM-L6-v2') client = chromadb.PersistentClient(path='./chroma_db') collection = client.get_collection('policy_docs') result = retrieve_and_answer( query = "Who approves leave without pay?", collection = collection, model = model, llm_call = call_llm, threshold = 0.6 ) print(f"Answer: {result['answer']}") for c in result['cited_chunks']: print(f" Source: {c['doc']} chunk {c['chunk']} score={c['score']}") # Output: # Answer: LWP requires approval from the Department Head AND # the HR Director. Manager approval alone is not sufficient. # [Source: policy_hr_leave.txt, chunk 11, score=0.84]

Protocol
What MCP is, how it works, and why the tool description is the Enforcement
The Problem

Why Agents Need a Standard Tool Interface

The integration problem

N agents × M tools = N×M custom integrations. Every agent must know the exact API of every tool it might call. It does not scale.

The discovery problem

An agent cannot ask "what tools exist?" without a standard protocol. Tool availability must be hardcoded into every agent prompt — every time.

MCP solves both

A standard: MCP servers expose tools. MCP clients discover and call them. Any agent can call any MCP server without custom integration code.

"MCP is to AI tools what HTTP is to web APIs — a standard that makes every tool callable by every client."
Architecture

MCP Server vs MCP Client

Two actors. One protocol. JSON-RPC 2.0 over HTTP POST.

The Protocol

Two JSON-RPC Methods

tools/list — discovery
// Request { "jsonrpc": "2.0", "method": "tools/list", "id": 1 } // Response { "result": { "tools": [{ "name": "query_policy_documents", "description": "...", ← Enforcement "inputSchema": { "type": "object", "properties": { "question": {"type":"string"} }, "required": ["question"] } }] } }
tools/call — execution
// Request { "jsonrpc": "2.0", "method": "tools/call", "params": { "name": "query_policy_documents", "arguments": { "question": "Who approves LWP?" } }, "id": 2 } // Response { "result": { "content": [{ "type": "text", "text": "Dept Head AND HR Director..." }], "isError": false } }
The Key Insight

Tool Description Is the Enforcement

The description is what the agent reads to decide when to call this tool — and when not to.

❌ Vague — agent calls for anything
"description": "Answers questions
about company policies." → Agent sends budget forecast question → Tool refuses (not in documents) → Agent wasted a call → In a chain: entire flow breaks
✅ Specific — agent routes correctly
"description": "Answers questions about
CMC HR Leave Policy, IT
Acceptable Use Policy, and
Finance Reimbursement Policy
ONLY. Returns refusal for
questions outside these three
documents." → Agent knows exact scope → Budget question: not sent here → Zero wasted calls
"This is the E in R.I.C.E — applied at the tool interface layer. Same principle. Different layer of the stack."
