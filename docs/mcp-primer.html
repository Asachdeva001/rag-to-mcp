Slide 1 of 6 · Context

Why agents need a standard tool interface

Without a standard — every tool requires custom integration

The integration problem
You have built a RAG server. Another team has built a database query tool. A third team has a calendar tool. Every agent that wants to use all three needs to know the exact API of each one. Three tools × N agents = N×3 custom integrations.
The discovery problem
An agent cannot ask "what tools are available?" without a standard. It has to be told explicitly what each tool does, what inputs it takes, and when to call it — hardcoded into the agent's prompt every time.
MCP: Model Context Protocol
A standard protocol: MCP servers expose tools. MCP clients (agents) discover and call them. Any agent can call any MCP server without custom integration. The tool description tells the agent what the tool does — and when not to use it.
"MCP is to AI tools what HTTP is to web APIs — a standard that makes every tool callable by every client."

Slide 3 of 6 · Protocol

The two JSON-RPC methods

tools/list — discover · tools/call — execute

→ tools/list request
{
  "jsonrpc": "2.0",
  "method": "tools/list",
  "id": 1
}
← tools/list response
{
  "result": {
    "tools": [{
      "name": "query_policy_documents",
      "description": "...", ← this is the enforcement
      "inputSchema": { ... }
    }]
  }
}
→ tools/call request
{
  "method": "tools/call",
  "params": {
    "name": "query_policy_documents",
    "arguments": {
      "question": "Who approves LWP?"
    }
  }
}
← tools/call response
{
  "result": {
    "content": [{
      "type": "text",
      "text": "Dept Head AND HR Director..."
    }],
    "isError": false
  }
}
"All responses return HTTP 200. Application errors use isError: true in the JSON body — not HTTP error codes. Unknown methods return JSON-RPC error code -32601."
Slide 4 of 6 · The Failure Mode

The tool description is the enforcement

Vague description → agent calls tool for anything · Specific → agent knows scope

❌ Vague description
"Answers questions about company policies."
Agent sends: "What is the 2025 budget forecast for Ward 3?"
Tool refuses. Agent wasted a call.
In a multi-step chain — this breaks the entire flow.
Agent had no way to know this question was out of scope.
✅ Specific description
"Answers questions about CMC HR Leave Policy, IT Acceptable Use Policy, and Finance Reimbursement Policy only. Returns cited answers grounded in retrieved document chunks. Returns a refusal for questions outside these three documents — do not send budget, procurement, or general policy questions to this tool."
Agent knows: only send HR / IT / Finance questions.
Budget question → agent does not call this tool.
No wasted calls. No broken chains.
"This is the E in R.I.C.E — applied at the tool interface layer instead of the prompt layer. Same principle. Different layer of the stack."
Slide 5 of 6 · The Full Picture

Three UCs · Three layers · One framework

R.I.C.E Enforcement applied at every layer of the AI engineering stack

1
UC-0A
Prompt layer — R.I.C.E Enforcement rules define what the classifier can and cannot output. No fixed enum = taxonomy drift. No severity keywords = severity blindness.
Prompt
2
UC-RAG
Retrieval layer — Enforcement rules define chunking strategy, similarity threshold, and context grounding. Wrong chunking = boundary failure. No threshold = wrong retrieval. No grounding = context breach.
Retrieval
3
UC-MCP
Tool interface layer — Tool description defines what the agent is allowed to call this tool for. Vague description = wrong tool calls. Specific description = correct agent routing.
Tool interface
"UC-MCP calls UC-RAG. UC-RAG reads the policy docs. UC-0A validated the prompt engineering discipline that makes both trustworthy. One framework. Three layers."
